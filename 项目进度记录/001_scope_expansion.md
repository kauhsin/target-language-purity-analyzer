# 001 Scope Expansion: Intended-Variety Text

Date: 2026-07-18

## Summary

The project scope was refined after kickoff. The original framing focused mostly
on LLM-generated dialect text. The updated framing is broader:

```text
Explainable target-variety consistency analysis for generated and human-produced text.
```

This is not a redesign of the Version 1 model. It is a broader and more accurate
definition of the analysis object.

## Decision

The project should not be limited to sentences generated by large language
models.

Version 1 should analyze any sentence that has an intended target language
variety, including:

- LLM-generated intended-variety text.
- Learner-produced intended-variety text.
- Human-written intended-variety text, including native-speaker or heritage
  speaker writing influenced by another written standard.
- Synthetic or manually constructed challenge examples.

The input source can vary, but the core task remains:

```text
Given a target variety and a sentence, analyze target-variety consistency and
competing-variety evidence.
```

## Rationale

The original LLM-focused framing was useful but too narrow. The same model and
explanation framework can apply to many texts where the writer or generator
intended to produce a specific variety.

For example, a sentence intended as Shanghainese may contain:

- Mandarin-influenced written forms because the writer was educated primarily in
  Mandarin.
- Cantonese-influenced forms because the writer is learning multiple Sinitic
  varieties.
- Mixed forms because a language model generated an unstable target-variety
  output.

In all cases, the analytical question is the same:

```text
Which spans provide evidence for the target variety, and which spans provide
evidence for competing varieties?
```

## What Changes

The project wording changes from:

```text
Analyze whether LLM-generated dialect text is contaminated.
```

to:

```text
Analyze whether text written or generated with an intended target variety shows
evidence for the target variety or competing varieties.
```

This affects project positioning, Milestone 0 problem definition, evaluation
framing, and future documentation.

## What Does Not Change

The accepted Version 1 technical design remains unchanged:

- Train one three-way Transformer classifier.
- Use pure Shanghainese, Cantonese, and Mandarin corpora for primary training.
- Do not add a `Mixed` class to the main classifier.
- Do not train a second mixed-language detector.
- Do not require token-level language labels.
- Use occlusion / leave-one-out attribution for explanations.
- Phrase outputs as evidence for each language variety.
- Do not claim absolute word-level language labels.
- Do not treat softmax scores as purity percentages.

## Version 1 Boundary

Even with the broader input scope, Version 1 remains a diagnostic and
explanation system. It should not promise:

- Automatic correction.
- Pedagogical tutoring.
- Semantic equivalence checking.
- Full sentence rewriting.
- Guaranteed target-variety replacement suggestions.

The safer Version 1 output remains:

```text
This span provides evidence for Mandarin in the model's decision.
```

not:

```text
This word is wrong and must be replaced with a specific target-variety form.
```

## Later-Version Direction

A later version may explore cautious candidate replacement suggestions if
additional semantic resources or parallel corpora are available.

One possible future workflow:

```text
Detect competing-variety evidence
→ Find a possible semantic bridge or aligned meaning
→ Retrieve candidate target-variety alternatives
→ Substitute candidate span
→ Re-check the sentence with the Version 1 analyzer
→ Present the result as a cautious candidate, not a guaranteed correction
```

This later direction should remain separate from Version 1. Version 1 must first
establish reliable detection, attribution, and evaluation.

## Next Milestone Implication

Milestone 0 should define the problem around:

```text
intended target variety
```

rather than:

```text
LLM-generated target dialect text only
```

Milestone 0 should explicitly define possible input-source categories while
keeping the Version 1 model task narrow and testable.
